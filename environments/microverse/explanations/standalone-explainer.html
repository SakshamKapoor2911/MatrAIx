Memory system and one agent, explained with two graphs

This page uses two simple visual flows and plain words to show how one agent in MircoVerse remembers, retrieves, acts, and sometimes reflects. The point is to make the system easier to understand without losing the real repo structure.

Working memory: observation packet Long-term memory: agent_memory Identity: original_soul vs current_identity Retrieval: read_memory + search_memory

Graph 1

Memory system

The memory system has three layers: current working context, authored long-term notes, and identity history. The agent reads the compact index first, then decides if it needs more.

Graph 2

Agent loop

One agent follows a tight loop: observe, judge memory, retrieve if needed, choose one action, and optionally reflect when the experience is strong enough to justify a change.

Plain-language memory guide

A
Working memory is the current turn. The server sends a fresh observation each tick. It is the agent's immediate context, not a long archive.
B
Long-term memory is authored by the agent. The agent can append or update notes in agent_memory. The engine stores what the agent chose to remember; it does not invent the note for it.
C
Memory is typed. Entries are organized as events, relationships, and reflections, which keeps subjective memory readable instead of one giant blob.
D
Retrieval is index-first. The agent sees a compact memory index and judges relevance itself. It only pulls full entries when a decision needs more detail.

Plain-language agent guide

One agent's loop is simple on purpose. It looks at the world, checks its memory index, maybe reads a few memories, and then takes exactly one action.

The important part is that action and memory stay connected. What the agent remembers, what it says it intends, and what it actually does are all logged separately so the experiment can study drift instead of hiding it.

Reflection is optional. The agent only revises its identity when it genuinely thinks the new self is justified by experience.

Where the code lives

Schema

mircoverse/persistence/schema.sql stores agents, agent_memory, and identity_snapshots.

DAL

mircoverse/persistence/dal.py writes memory deltas and builds the memory index.

Agent tools

mircoverse/agents/tools.py exposes read_memory, search_memory, submit_action, and submit_reflection.

Turn prompt

mircoverse/agents/prompts.py renders the observation and memory index for the agent.

Short summary

The memory system is the agent's visible context plus its authored notes plus its identity log. The agentic system is the repeated loop that reads that context, decides, acts, and sometimes reflects.

The point of separating them is analytical clarity: you can tell what happened, what the agent chose to keep, and how it later chose to define itself.
